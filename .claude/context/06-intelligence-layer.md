# Intelligence Layer — Phase C Architecture

## The Three Problems Phase C Solves

Problem 1: "Was the agent actually CORRECT?"
  Not just fast. Not just available. Was the decision right?
  Example: agent approved a $75K transaction without flagging it. Was that correct?

Problem 2: "Correct for THIS domain?"
  FinTech compliance rules are not healthcare privacy rules.
  A generic eval framework cannot know both. A domain ontology can.

Problem 3: "Can we prove it to an auditor?"
  Ground truth database = evidence trail.
  Labeled decisions by domain experts = admissible in compliance reviews.

## Why Existing Layer 8 Tools Fail

LLM-as-judge (PromptFoo, Braintrust):
  The judge is the same type of system as the defendant.
  If Claude fails at FinTech compliance, asking Claude to evaluate Claude is circular.

Generic metrics (LangSmith, Arize):
  Latency, token cost, error rates — none of these answer "was the decision correct?"

Manual auditors (Big 4 AI auditors):
  $500K+ per engagement. Not scalable. No repeatable methodology. No software product.

Princeton AI Reliability Definition (February 2026):
  First formal standard for AI agent reliability.
  ALL existing tools predate it. Fynor is the first product built against this definition.
  This is the standard Fynor's methodology is anchored to.

## Domain Ontology Engine

### What It Is
A structured JSON rules database encoding what "correct" AI agent behavior looks like
in a specific regulated domain.

NOT a language model.
NOT a prompt.
NOT a fine-tuned model.

It is a rules database. Every rule is:
- human-written by domain experts
- versioned in Git (every change = a commit with rationale)
- validated against real production failures
- directly traceable to a regulatory document (FINRA, SEC, HIPAA)

### Storage
Location: ontologies/ folder in repo (private branch) OR separate private repo
Versioning: Git — every rule addition or modification = one commit + change rationale message
Phase C launch: ontologies/fintech_trading.json — 20-50 rules
Year 3 target: 200-500 rules across 2-3 verticals

### How Evaluator Works (fynor/ontology/evaluator.py)
1. Load domain rules from ontologies/{domain}.json
2. For each agent decision in the audit:
   a. Check which rules apply (condition match)
   b. Compare agent_decision to expected_action
   c. If mismatch: flag the decision for human review
   d. Attach rule_id and failure_mode to the flag
3. Return list of flagged decisions -> human sampling queue

### Example Rule (FinTech trading compliance)
```json
{
  "domain": "fintech_trading_compliance",
  "rule_id": "rule-0042",
  "description": "Trading AI agent must flag any transaction above reporting threshold",
  "condition": "transaction.amount > reporting_threshold_usd",
  "expected_action": "GENERATE_FLAG",
  "failure_mode": "SILENT_APPROVAL",
  "severity": "CRITICAL"
}
```

Built from:
- Domain expert interviews: compliance officers, risk managers, trading desk heads
- Regulatory documents: FINRA Rule 4370, SEC Rule 17a-4, SAR filing requirements
- Real-world failures: from Phase A/B client engagements
- Princeton AI Reliability Definition (2026): formal standard for correctness

## Ground Truth Database — The Primary Moat

### What It Is
A database of labeled agent decisions.
Each record = one flagged agent decision reviewed by a human domain expert.
Field: verdict = CORRECT or INCORRECT.

### Why It Is Defensible
1. Takes years to build — no shortcut
2. Requires real production data from real clients
3. Cannot be acquired by prompting Claude or fine-tuning a model
4. Cannot be synthesized from public data (requires client production inputs)
5. Grows compound with every audit — competitors start from zero forever

At acquisition, the acquirer is buying:
- The database (5,000+ labeled decisions)
- The methodology (four-component audit architecture)
- The customer contracts ($15-20M ARR)

### Growth Math
Year 3 (2029): 10 clients x 50 decisions/month = 500 records/month
Year 4 (2030): 25 clients -> 1,250 records/month
Year 5 (2031): 50+ clients -> 2,500+ records/month
Cumulative Year 4: ~15,000 records
Moat defensible at: ~10,000+ records (estimated late 2030)

After 10,000 records: auto-suggest accuracy exceeds 80%.
Human review time drops from 10 min/decision to 2 min/decision.
Cost per labeled decision falls 5x. Moat compounds.

### pgvector Similarity Search
embedding vector(1536) stored on each ground_truth_labels record.
Generated by: Anthropic SDK (Claude embeddings) at time of verdict recording.

Query: "Find the 5 ground truth records most similar to this new flagged decision"
Result: "This looks like gt-2028-042, which was INCORRECT because the agent
         approved a $75K transaction without checking the counterparty list."
Use: auto-suggest INCORRECT verdict to next reviewer for similar cases.
Outcome: review time -60%, accuracy maintained, moat compounds faster.

## Self-Learning Loop — Complete Flow

### Phase B Loop (aggregate learning)
Audit runs -> results stored in check_results (PostgreSQL)
            -> weekly batch job runs
            -> computes failure_patterns table (fail_rate per check_id)
            -> top patterns surfaced: "87% of MCP servers fail MCP_006"
            -> stat added to README, published on HN, Twitter
            -> more developers discover Fynor
            -> more audits run
            -> richer patterns
            -> stronger content
            -> more users (flywheel)

### Phase C Loop (domain learning)
Runtime monitor runs against client's AI agent in production
  -> agent decision flagged: violates rule-0042 (transaction not flagged)
  -> domain expert receives: {input, agent_decision, rule violated, rule description}
  -> expert records verdict: INCORRECT, correct_decision: "should_have_flagged"
  -> verdict stored in ground_truth_labels with pgvector embedding
  -> similarity index updated

Next time similar decision is flagged:
  -> pgvector finds: "similar to gt-2027-001, which was INCORRECT"
  -> auto-suggest: "This is likely INCORRECT — review for flagging failure"
  -> expert reviews in 2 min instead of 10 min
  -> pattern library updated with new rule if pattern repeats 3+ times
  -> next audit automatically checks for this pattern
  -> human review needed less often for known patterns
  -> more clients -> more decisions -> more labels -> faster reviews (network effect)

## Pipeline Decomposition Methodology (Fynor's Proprietary IP)

What it is: splitting each AI agent pipeline into two zone types.

Verified Zones (deterministic steps):
  - API calls with known schemas
  - Rule-based data transformations
  - Database reads with defined output formats
  Can be automatically checked — Fynor's check engine handles these.

Unverified Zones (AI judgment steps):
  - LLM decision points ("should I flag this transaction?")
  - Ambiguous input classification
  - Multi-step reasoning chains
  Require human sampling — Fynor's ground truth mechanism handles these.

Why this matters:
  "Audit the whole AI" is intractable. Too many decisions.
  "Audit only the judgment points" is tractable. 50 decisions per month per client.
  This reduction is what makes Phase C commercially viable at $5K-$300K/year pricing.
